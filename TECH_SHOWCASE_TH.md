# เบื้องหลังการทำ HueMind (Tech Insights & Architecture)

เอกสารนี้รวบรวมเทคนิคและวิธีการแก้ปัญหาต่างๆ ตอนที่ทำโปรเจกต์ **HueMind** เอาไว้อ่านทบทวน หรือเอาไว้เล่าให้เพื่อนฟังเล่นๆ ว่าตอนทำเราแอบยัดเทคนิคอะไรไปบ้างครับ

---

## 1. ระบบ Real-time แบบเบาบางด้วย SSE (Server-Sent Events)
ปกติถ้าพูดถึงการทำเกม Multiplayer เรามักจะนึกถึง WebSocket สมัยหน้ากาก แต่ด้วยความที่ตัวเกมของเราไม่ได้มีการอัปเดต state ถี่ขนาดระดับเสี้ยววินาที (เช่น เกมเดินยิงหรือแอคชัน) การเปิด WebSocket แบบ Two-way อาจจะกินทรัพยากรเกินความจำเป็น

เราก็เลยเลือกใช้เทคนิค **Server-Sent Events (SSE)** แทนครับ หลักการคือฝั่ง Client จะยิง HTTP Request ไปเปิดทิ้งไว้ แล้ว Server คอยสตรีมข้อมูลทางเดียวส่งกลับมาเรื่อยๆ
- **ข้อดี:** กินทรัพยากรน้อยกว่ามาก ฝั่งเบราว์เซอร์จัดการเรื่อง Auto-reconnect ให้เองได้ด้วย ทำให้สถาปัตยกรรมเกมของเราเบาและเสถียรพอดีกับสเกลตัวเกมเลย

## 2. การจัดการ Performance (ลด React Re-render ลำบากเกิ้น)
ตอนแรกที่ทำระบบมาเจอปัญหาหนักอยู่อย่างนึงคือ **กระดานเกมมันมีตั้ง 480 ช่อง** (30x16)
ปัญหาเกิดตอนเราเพิ่มช่องแชทเข้าไป เวลาผู้เล่นพิมพ์ข้อความรัวๆ React มันดันพยายามจะไป Re-render หน้าจอใหม่ทั้งหมด รวมถึงไอ้ตารางสี 480 ช่องนั่นด้วย ผลคือแค่พิมพ์ข้อความเว็บก็แอบหน่วงแล้ว

เราเลยแก้ด้วยการนำ **`React.useMemo`** มาห่อครอบตัวตารางเกมทั้งหมด แล้วระบุ dependency ไว้ชัดเจนว่า "ให้วาดตารางใหม่เฉพาะตอนที่เป้าหมายเปลี่ยน หรือมีการกดทายสีเท่านั้นนะ"
- **ข้อดี:** พอแยกส่วนปุ๊บ พิมพ์แชทรัวแค่ไหน ตารางสีก็ไม่ถูกดึงไปวาดใหม่ หน้าเกมเลยกลับมาใช้ลื่นๆ พิมพ์ลื่นเหมือนแชทปกติเลย

## 3. คณิตศาสตร์เบื้องหลังระบบนับแต้ม (Chebyshev Distance)
ใน Board Game ของจริง เวลาให้แต้มมันจะนับระยะห่างเป็นช่องๆ รอบทิศ กางออกมาเป็นสี่เหลี่ยม (รอบใน 3x3 ย่อย, รอบนอก 5x5 ทับอีกชั้น)
ปัญหาคือถ้าเราใช้สมการหาระยะห่างแบบลากเส้นตรงทะแยงมุมปกติ (Euclidean distance หรือ a² + b² = c²) วงกว้างมันจะกางออกเป็นวงกลม ซึ่งคนทายตกตามมุมสี่เหลี่ยมจะอดได้คะแนน

เราเลยนึกถึงสูตร **Chebyshev Distance** ขึ้นมา (หลักการเหมือนระยะการเดินของตัว King ในหมากรุก)
โดยใช้สมการเช็คว่าแกนไหนห่างมากสุด: `ระยะห่าง = Math.max( |x1 - x2|, |y1 - y2| )`
- **ผลลัพธ์:** แค่นี้เราก็ได้ฟังก์ชันหน้าตาคลีนๆ ที่กางรัศมีเป็นรูปสี่เหลี่ยม 3x3 หรือ 5x5 เป๊ะๆ เหมือนเล่นบนกระดานจริงแล้วครับไม่ต้อง hardcode เลย

## 4. วิธีคิดของ AI Bot (Heuristic Rule-Based)
บอทในเกมเราไม่ได้ใช้โมเดล AI ลึกล้ำอะไรครับ เราเขียนโลจิกให้ตัดสินใจแบบ Heuristic คล้ายๆ วิธีคิดของคน 
- ถ้าระดับง่าย (Easy) บอทมันก็จะหาช่องสุ่มไปเรื่อยๆ ในรัศมีกว้างๆ (+- 12 ช่อง)
- ถ้าระดับนรก (Very Hard) มันจะบีบเป้าไปทายห่างจากเฉลยจริงๆ ในระยะแค่ 1 ช่อง
- ความเจ๋งคือเราเผื่อระบบ **Fallback (Spiral Scanning)** ไว้ด้วย สมมติในโหมดยากแล้วช่องใกล้เคียงเป้าหมายมันโดนคนอื่นจิ้มไปหมดแล้ว (กรณีที่มีตั้งค่าห้ามทายช่องซ้ำ) บอทมันจะสแกนหาช่องว่างที่ใกล้ที่สุดถัดออกไปแทนอัตโนมัติ

## 5. ระบบจัดอันดับร่วม (Standard Competition Ranking)
ตอนแรกระบบสรุปกระดานผู้นำ (Leaderboard) มันเรียง 1, 2, 3 ตามปกติ 
แต่ในบอร์ดเกมจริงๆ ถ้ายอดคะแนนสองคนมันเท่ากัน มันต้องถือว่าครองอันดับ 1 ร่วมกัน เราเลยแก้ระบบจัดอันดับให้มาใช้ลอจิกคล้ายกีฬาสากล (อย่างกอล์ฟ)
คือถ้าคนแรกดึงคะแนนที่ 50 (ได้ที่ 1) และคนที่สองก็ได้ 50 ระบบจะโชว์ว่า "อันดับ 1" ทั้งคู่ ส่วนคนที่สามได้คะแนน 48 จะโดนปัดลงไปโชว์ "อันดับ 3" เลยให้อารมณ์การแข่งมันตึงแบบเพื่อนฝูงกว่าเดิม

## 6. โครงสร้างเบื้องหลัง In-Memory & TypeScript
- เพื่อให้เกมตอบสนองไวที่สุด เราตั้งใจตัด external database ออกไปก่อน แล้วเก็บ State ทุกอย่างบน RAM (In-Memory `Map`) หมดเลย ความหน่วง (Latency) เลยแทบเป็นศูนย์
- โค้ดทั้งหมด Migrate จาก JavaScript เป็น **TypeScript 100%** วาง `Interface` ชัดเจน (`Room`, `Player`, `GameState`) ทำให้เวลาเราเพิ่มฟีเจอร์ โค้ดจะเช็ค type และดักปัญหาให้หมดตั้งแต่ตอนเขียน แทบไม่เจอพวกรันไทม์แครชโง่ๆ เลยเวลาเล่นจริง.

## 📖 ภาคผนวก: รวมความหมายคำศัพท์ฉบับรวบรัด (Glossary)

เพื่อให้อธิบายให้เพื่อนฟังได้เคลียร์ขึ้นเผื่อโดนถามกลับ นี่คือความหมายแบบภาษาคนของศัพท์ที่โผล่มาในเอกสารครับ:

- **SSE (Server-Sent Events):** ระบบส่งข้อมูลทางเดียวจากเซิร์ฟเวอร์มาที่เบราว์เซอร์ เหมาะกับงานที่ไม่ต้องส่งกลับไปถี่ๆ ประหยัดเน็ตและเครื่องเซิร์ฟ
- **WebSocket (Two-way Communication):** ช่องทางเชื่อมต่อแบบไป-กลับ เปิดท่อส่งข้อมูลชนกันตลอดเวลา เหมาะกับแอปแชทเรียลไทม์ หรือเกมรถแข่ง/เดินยิง
- **State:** ตัวแปรที่เก็บ "สถานะ" ปัจจุบันของแอป เช่น ตอนนี้เกมอยู่รอบที่เท่าไหร่ ใครเป็นคนใบ้ แชทมีข้อความอะไรบ้าง
- **Re-render (รีเรนเดอร์):** การที่โค้ด React พยายามจะวาดหน้าเว็บขึ้นมาใหม่ หรือเอาสีไปป้ายใหม่บนจอ
- **React.useMemo:** คำสั่งให้ React "จำ" ขอบเขตหรือผลลัพธ์บางอย่างไว้ ถ้าข้อมูลไม่มีอะไรใหม่ ก็ไม่ต้องลงแรงไปวาดมันใหม่ (Memoize/จำเอาไว้)
- **Dependency:** เงื่อนไขผูกมัดหรือตัวแปรควบคุม (ในเคส useMemo คือการบอกว่า ถ้าตัวแปรพวกนี้ไม่เปลี่ยน ก็ไม่ต้องวาดตารางสีใหม่นะ)
- **Heuristic (ฮิวริสติก):** การเขียนเงื่อนไขให้บอทคำนวณแบบอาศัย "กฎเกณฑ์คร่าวๆ / กะเอา" ตามประสบการณ์ ไม่ต้องสมบูรณ์แบบ 100% ประมวลผลไวแบบคนคิด
- **Fallback:** แผนสำรอง! เวลาเกิดข้อผิดพลาดหรือทำงานตามแผนเอไม่ได้ โค้ดจะไหลไปทำแผนบีแทนไม่ให้ระบบพัง (เช่น ถ้าช่องเต็ม ก็ให้ไปหาช่องใกล้ๆ แทน)
- **Spiral Scanning (สแกนแบบก้นหอย):** วิธีวนลูปค้นหาข้อมูลจากจุดศูนย์กลาง แล้วหมุนหาจุดว่างๆ กว้างออกไปทีละชั้น เหมือนลายยากันยุง
- **In-Memory:** การเอาข้อมูลไปฝากเก็บไว้บน RAM ของ Server แทนที่จะไปเอาลง Database ทั่วไป ข้อดีคือโคตรเร็วชนิดสับนิ้ว แต่ถ้าเครื่องดับข้อมูลก็จะหายไป (เหมาะกับ Session ของเกมกระดานแบบนี้มากๆ)
- **Latency:** ความหน่วง หรือปิง ยิ่งสูงแปลว่าเกมสั่งแล้วตอบสนองช้า
- **TypeScript / Interface:** ภาษาเขียนโค้ดที่บังคับให้ระบุ "ชนิดข้อมูล (Type)" ชัดเจน เช่น ต้องเป็นตัวเลขเท่านั้น (Interface คือการสร้างแม่พิมพ์จำลองข้อมูลล่วงหน้า)
- **Runtime Crash:** อาการเวลาเล่นหรือกดใช้งานอยู่ดีๆ แล้วหน้าจอระเบิด แหลกสลาย หน้าขาวไปเลย เพราะโค้ดดึงข้อมูลผิดประเภทกลางคัน
